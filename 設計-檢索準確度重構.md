# AI 值班輔助系統 — 檢索準確度重構設計

## Context（為什麼做這件事）

值班團隊面對 user 報案（JIRA ITSM，累積兩年資料），需要快速找出解法。現況已有一套 RAG 系統：把過往資料整理成**關鍵字 index**，搭配 TKMS 的 SOP 提供給值班；UI 上有「採用 / 否決」按鈕，否決時請值班補充，並有 review 功能驗證答對與否。

**核心痛點：準確度不夠高，且最常見的失敗型態是「完全找錯」**——撈回來的是不相干的案子。

根因判斷：對「模糊、用詞不一致」的文字報案，**關鍵字檢索語意對不上**，因此召回大量無關案件。本案不是從零開發，而是**針對檢索層的準確度重構**。

### 已釐清的前提

- **第一版範圍**：非 pipeline 的文字類報案（權限、環境、咨詢等）。pipeline 失敗類（有 URL、可串 Nexus/Harbor/Azure）留待後續。
- **知識來源權重**：**歷史已解案為主、SOP 為輔**。很多解法是值班的隱性經驗，只存在於過往單子、不在任何 SOP。
- **不確定系統時的行為**：AI 提示值班「該追問 user 什麼」，而非硬猜。
- **使用介面**：獨立 Web 工具（值班貼上 ticket 編號/內文 → AI 回建議），不動 JIRA。
- **成功指標（北極星）**：找得準（檢索命中品質）。
- **資料存取**：JIRA 與 TKMS 的取得路徑使用者已有，本設計不涵蓋。
- **回饋飛輪**：採用/否決/補充/review 機制已存在，可作為評測與調校燃料。

## 重構架構（五層，由痛點往外）

### 1. 查詢理解層（新增，最高優先）
值班貼入的 ticket 內文雜亂模糊，且**輸入品質本身常是「完全找錯」的最上游元兇**（爛查詢進 = 爛結果出）。先用 LLM 將其整理成結構化查詢：
- 抽出症狀關鍵詞、實體（服務 / 錯誤碼 / 環境）、可能系統、同義詞擴展。
- **輸入充足度檢查（必備、優先做）**：先判斷此 ticket 的資訊量是否足以支撐檢索。不足時**不硬撈**（硬撈必然找錯），直接輸出「建議追問 user 的問題清單」，待資訊補齊再檢索。這是不依賴任何進階技術、現在就能做的安全網，直接打中「完全找錯」的上游成因。
- **圖片理解（加分項、可後補）**：很多報案的關鍵錯誤訊息在截圖裡。若有具視覺能力的模型（如 Opus / Sonnet），讀 ticket 附圖 → 抽出錯誤訊息 / 畫面症狀，併入結構化查詢。
- **讀圖與追問解耦的降級邏輯**：
  - 讀圖能做 → 自動把截圖內容補進查詢。
  - 讀圖暫時做不到 → 不假裝看得懂、不硬撈。特別辨識「**有附圖但無法判讀內容**」的情況，明確請值班 / user 把圖裡的錯誤訊息貼成文字後再檢索。
  - 即使讀圖一直未上線，系統也不會因「資訊在圖裡」而亂找——由充足度檢查兜底。

### 2. 混合檢索（取代純關鍵字）
- 語意向量檢索（embedding）+ 關鍵字 BM25 同時召回（hybrid）。
- 語意層處理「用詞不同、意思相同」；關鍵字層處理錯誤碼、專有名詞等需精確命中者。
- 對兩個 corpus 各檢索：**歷史案 corpus 為主、SOP corpus 為輔**。
- Chunking 時將「問題描述」與「解法/回覆」分離標註，提升召回精度。

### 3. 重排序 Reranking（新增）
- 用 cross-encoder 或 LLM reranker 對候選做第二輪精排，把最該用的推到最前。
- 準確度導向的關鍵一步。

### 4. 生成 + 引用 + 信心
- RAG 產出回覆草稿，每段標清來源（哪張歷史單 / 哪份 SOP）與信心度。
- 信心低時改輸出「建議追問 user 的問題清單」，不硬掰。

### 5. 量測與飛輪（把既有回饋變燃料）
- 將現有採用/否決/補充/review 資料整理成**評測集（golden set）**。
- 每改一層都用它跑 recall@k / 命中率，**用數字證明變準**。
- 否決+補充作為負樣本，持續餵回調校 reranker 與查詢理解。

## 與現有版本的差異

| 環節 | 現況 | 重構後 |
|------|------|--------|
| 查詢 | 直接拿 ticket 內文（讀不到圖、太簡潔也照撈） | 新增 LLM 查詢理解層：結構化 + 同義詞擴展 + 讀截圖 + 輸入充足度檢查 |
| 檢索 | 關鍵字 index | 語意 + 關鍵字混合檢索 |
| 排序 | 無 | 新增 reranking 精排 |
| 生成 | 提供建議 | 附引用來源 + 信心度；低信心改追問 |
| 量測 | 採用/否決/review（觀察用） | 轉成 golden set，驅動可量化的迭代 |

## 實作順序（建議）

1. **先建評測集（第 5 層）**——讓後續每個改動都可被驗證；否則仍是「感覺不準但不知道改哪」。
2. 查詢理解層的上游修正（第 1 層）——**先做輸入充足度檢查 + 追問**（必備、低成本、高回報，且不依賴讀圖）；讀圖為加分項，有資源再補。上游乾淨了，後面整條鏈都受惠。
3. 升級檢索為混合檢索（第 2 層）——直接打「完全找錯」的根因。
4. 查詢理解層的結構化/同義詞擴展補完（第 1 層其餘）。
5. 加 reranking（第 3 層）。
6. 強化生成的引用/信心/追問（第 4 層）。

每步都對照 golden set 量測，確認 recall@k / 命中率提升再進下一步。

## 驗證方式

- **評測集回歸**：每層改動前後跑同一份 golden set，比較 recall@k、命中率、否決率。
- **A/B 對照**：新檢索流程 vs 現有關鍵字流程，跑歷史案件重放，比較「該找的有沒有被找到」。
- **值班實測**：上線後追蹤採用率 / 否決率 / 平均處理時間是否改善。

- 知識飛輪自動回寫（使用者現有機制可整併進來）。
