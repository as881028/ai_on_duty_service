# 迭代日誌

本 repo 由「大腦 agent」持續迭代：每輪派 3 個 sub agent 從不同視角找當前痛點 → 評估收斂 → commit & push。記錄每輪採納/否決的決策，直到擠不出更好的做法為止。

---

## 第 1 輪（2026-06-24）

**3 個 sub agent 視角**：① 檢索準確度技術缺口 ② prompt 可執行性 ③ 真實失敗模式/維運。

**採納（寫入 v2）**：
1. 🔴 資料治理前置（敏感資訊遮罩）→ 新增「階段 0a」+ 設計文件治理章節 A。JIRA 單含帳密/個資，補晚 corpus 要重洗。
2. 🔴 困難負例標註 + 「陷阱單擠進 top-1/3 比率」對抗指標 → 階段 0。只測 Recall@k 量不到「自信地找錯」。
3. 🔴 query/document 對稱萃取 + chunk 粒度上限 → 階段 2。「以問搜問」兩端萃取需對稱否則破功。
4. 🔴 失效解法處理（時效污染提前 + 人工失效黑名單）→ 階段 2 + 治理章節 B。系統越準、散播錯誤解法越快。
5. 🔴 回饋飛輪標註去偏（強/弱信號、位置偏差、否決分因）→ 階段 4 + 治理章節 C。
6. 🔴 驗收門檻無數字的自我矛盾 → 執行原則加「門檻產生機制 + `【待填】` 參數」。
7. 🟠 階段1門檻基於現有關鍵字分數、pipeline 最小判定規則、golden set 算術矛盾(270 vs 300-500)+缺料退路、順序權威聲明、漂移監控常設、延遲/成本 SLA + 可降級。

**否決（避免過度設計）**：HyDE / multi-query、knowledge graph、自建 fine-tune embedding。理由：第一版「文字類找相似單」範圍內屬邊際遞減或殺雞用牛刀；Agent 1 自己也標為不建議。

**收斂判斷**：未收斂。第一輪缺口多且高價值，3 視角互補幾無重複。續跑下一輪。

## 第 2 輪（2026-06-24）

**批判對象**：v2。3 agent 均明確表態「接近收斂」，但各挖到真正高價值的剩餘缺口。

**採納（寫入 v3）**：
1. 🔴 召回覆蓋率保底（A1）→ 階段 2 加 multi-query fan-out（復用既有 LLM）+ 原文 BM25 並聯保命路；階段 0 加「正解在 top-50 比率」召回上限指標。v2 所有防線都在召回「之後」，正解沒進候選集就回天乏術。
2. 🔴 複核品質保證（A3）→ 階段 0 加雙人盲覆核（難/跨版本格）、「未復發」明確定義觀察窗、每季 5% 複核的複核。整套量測建立在值班複核單點，錯誤被數字化反而有權威感。
3. 🟠 對稱萃取單點故障（A1）→ 原文 BM25 並聯（同上）；trace 記萃取後症狀句。
4. 🟠 時序矛盾（A2）→ 明確 query 理解含「結構化症狀萃取子任務」、階段 1 落地、schema 凍結後階段 2 複用。
5. 🟠 術語統一（A2）→ 開頭加名詞統一（陷阱單=hard negative=同池，三元組格式）；對抗指標分母=難級子集。
6. 🟠 服務改名 alias map（A1+A3）→ 階段 2 加 alias map 掛兩端 + 詞典從 CI/CD 平台清單匯入 + regex 保護強結構 token；階段 4 加實體分布漂移偵測。
7. 🟡 萃取版本管理（extraction_model/prompt_version 比對告警回填）、pipeline 邊界（僅掃文字+被擋日誌）、0a 抽樣量具體化+待填時機。

**否決（過度設計）**：HNSW/ANN 調參、自訓 embedding/reranker、learning-to-rank、HyDE/多輪對話檢索、災備。3 agent 均自行標為此規模下反模式。

**收斂判斷**：接近收斂但未到。本輪仍出現 2 個 🔴（召回保底、複核品質），屬 v2 防線的真實盲區。3 agent 一致表示「補完這幾項即收斂」。續跑第 3 輪驗證是否真的擠不出高價值項。

## 第 3 輪（2026-06-24，收斂驗證）

**做法**：門檻拉到最高，只接受「不補就明顯拖累準確率 / 會被迫下線」的硬傷。

**結果**：**2/3 agent 判定「已收斂」**（檢索技術、可執行性）。維運 agent 抓到最後 1 個達門檻硬傷。

**採納（v3 微調）**：
1. 🔴 **失效黑名單治理**（A3）→ 階段 2.5 補：標記須附 ID+理由+生效範圍；爭議走難格雙人盲覆核同一裁決動線；黑名單增刪當「corpus 變更事件」納漂移對賬。理由：黑名單是繞過所有數字 gate 的人工旁路，「命中即硬排除」誤標會把高頻正解整批逐出候選集，CI/漂移監控抓不到，直接製造完全找錯並打穿採用率曲線。

**否決/確認不達門檻**：multi-query 成本放大、reranker 降級告警、版本回填 race、低信心格、pipeline 截圖漏接——皆已被現有段落覆蓋或屬「上線後再優化」。

**收斂判斷**：實質收斂。3 視角中 2 個已明確收斂，第 3 個的唯一硬傷已補。續跑第 4 輪做最終確認；若全數收斂即停止迴圈。

---

## 第 4 輪（2026-06-24，最終確認）

**結果**：**3/3 agent 全數判定「已收斂」**，無任何達門檻的新項。
- 檢索技術：量測地基、召回保底、對稱、失效旁路、alias map、飛輪去偏、範圍邊界全齊；唯一可挑的「回填期暫態不對稱」已有原文 BM25 兜底，不達門檻。
- 可執行性：逐行確認黑名單治理段與既有機制引用一致（裁決動線、漂移對賬掛得上）、無矛盾、篇幅可控。
- 維運：上輪硬傷三個失血口（誤標/爭議/不可觀測）已封死；其餘隱患逐項對照皆已收斂；剩餘僅 `【待填】` 數字門檻，屬流程約束非設計缺陷。

## 迴圈終止（收斂達成）

連續第 4 輪後，3 個視角的 sub agent 一致「擠不出更好的做法」，達成使用者設定的終止條件。

**最終產物**：`執行prompt-檢索準確度重構.md`（v3）為可直接交付的執行 prompt，覆蓋 0a 資料治理 → 0 量測地基 → 1 止血 → 2 混合檢索 → 3 精排 → 4 飛輪 + 維運/信任，所有關鍵閾值綁定 golden set sweep、`【待填】` 參數待與使用者敲定。

**迭代統計**：4 輪 × 3 agent = 12 次審查；採納高價值改善約 20 項（第1輪7、第2輪7、第3輪1，含合併）；一致否決過度設計項（HyDE/KG/自訓模型/LTR/HNSW 調參/災備等）。

## 附篇（2026-06-25）：人員協作維度

系統面收斂後，使用者提出新維度——**「人」（值班、管理員、主管）能做什麼幫系統更準**。沿用同一流程派 3 個 sub agent，分別從「第一線值班」「知識維護/管理員」「主管/管理層」三個角色視角構思。

**整合產物**：新增 `人員協作指南-提升值班準確度.md`，按三角色給出「該做/不該做」並對應到設計機制。

**最有價值的訊號——三視角獨立收斂的四條紅線**：
1. 採用率永遠不能當個人 KPI（只能當系統信任健康度曲線）。
2. 棄答/追問是正當行為，不是系統爛。
3. 否決必須分因（「找錯」才是 hard negative，「想自己處理」不是）。
4. 失效黑名單必須有 owner 與裁決動線（最強人工旁路、誤用代價最大）。

**核心觀念**：系統是放大器不是發電機——人餵什麼品質，系統放大什麼品質。工程把放大器調好，人決定餵什麼進去。

**一致排除的人為過度負擔**：要求值班手填結構化 schema / 做 PII 遮罩 / 標三元組；管理員逐筆人工檢查遮罩、每日跑指標、全量雙覆核、手調閾值；主管微觀審每筆黑名單、把一致性/讀圖時程 KPI 化、為棄答開檢討會。原則：機器能做的（偵測/記錄/跑分/遮罩/調參）交自動 pipeline，人只接異常告警並做「機器判斷不了的價值裁決」。
