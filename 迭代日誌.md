# 迭代日誌

本 repo 由「大腦 agent」持續迭代：每輪派 3 個 sub agent 從不同視角找當前痛點 → 評估收斂 → commit & push。記錄每輪採納/否決的決策，直到擠不出更好的做法為止。

---

## 第 1 輪（2026-06-24）

**3 個 sub agent 視角**：① 檢索準確度技術缺口 ② prompt 可執行性 ③ 真實失敗模式/維運。

**採納（寫入 v2）**：
1. 🔴 資料治理前置（敏感資訊遮罩）→ 新增「階段 0a」+ 設計文件治理章節 A。JIRA 單含帳密/個資，補晚 corpus 要重洗。
2. 🔴 困難負例標註 + 「陷阱單擠進 top-1/3 比率」對抗指標 → 階段 0。只測 Recall@k 量不到「自信地找錯」。
3. 🔴 query/document 對稱萃取 + chunk 粒度上限 → 階段 2。「以問搜問」兩端萃取需對稱否則破功。
4. 🔴 失效解法處理（時效污染提前 + 人工失效黑名單）→ 階段 2 + 治理章節 B。系統越準、散播錯誤解法越快。
5. 🔴 回饋飛輪標註去偏（強/弱信號、位置偏差、否決分因）→ 階段 4 + 治理章節 C。
6. 🔴 驗收門檻無數字的自我矛盾 → 執行原則加「門檻產生機制 + `【待填】` 參數」。
7. 🟠 階段1門檻基於現有關鍵字分數、pipeline 最小判定規則、golden set 算術矛盾(270 vs 300-500)+缺料退路、順序權威聲明、漂移監控常設、延遲/成本 SLA + 可降級。

**否決（避免過度設計）**：HyDE / multi-query、knowledge graph、自建 fine-tune embedding。理由：第一版「文字類找相似單」範圍內屬邊際遞減或殺雞用牛刀；Agent 1 自己也標為不建議。

**收斂判斷**：未收斂。第一輪缺口多且高價值，3 視角互補幾無重複。續跑下一輪。

## 第 2 輪（2026-06-24）

**批判對象**：v2。3 agent 均明確表態「接近收斂」，但各挖到真正高價值的剩餘缺口。

**採納（寫入 v3）**：
1. 🔴 召回覆蓋率保底（A1）→ 階段 2 加 multi-query fan-out（復用既有 LLM）+ 原文 BM25 並聯保命路；階段 0 加「正解在 top-50 比率」召回上限指標。v2 所有防線都在召回「之後」，正解沒進候選集就回天乏術。
2. 🔴 複核品質保證（A3）→ 階段 0 加雙人盲覆核（難/跨版本格）、「未復發」明確定義觀察窗、每季 5% 複核的複核。整套量測建立在值班複核單點，錯誤被數字化反而有權威感。
3. 🟠 對稱萃取單點故障（A1）→ 原文 BM25 並聯（同上）；trace 記萃取後症狀句。
4. 🟠 時序矛盾（A2）→ 明確 query 理解含「結構化症狀萃取子任務」、階段 1 落地、schema 凍結後階段 2 複用。
5. 🟠 術語統一（A2）→ 開頭加名詞統一（陷阱單=hard negative=同池，三元組格式）；對抗指標分母=難級子集。
6. 🟠 服務改名 alias map（A1+A3）→ 階段 2 加 alias map 掛兩端 + 詞典從 CI/CD 平台清單匯入 + regex 保護強結構 token；階段 4 加實體分布漂移偵測。
7. 🟡 萃取版本管理（extraction_model/prompt_version 比對告警回填）、pipeline 邊界（僅掃文字+被擋日誌）、0a 抽樣量具體化+待填時機。

**否決（過度設計）**：HNSW/ANN 調參、自訓 embedding/reranker、learning-to-rank、HyDE/多輪對話檢索、災備。3 agent 均自行標為此規模下反模式。

**收斂判斷**：接近收斂但未到。本輪仍出現 2 個 🔴（召回保底、複核品質），屬 v2 防線的真實盲區。3 agent 一致表示「補完這幾項即收斂」。續跑第 3 輪驗證是否真的擠不出高價值項。
