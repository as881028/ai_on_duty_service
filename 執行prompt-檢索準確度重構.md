# 任務 Prompt：AI 值班輔助系統 — 檢索準確度重構（v3）

> **名詞統一（避免歧義）**：「**陷阱單**」=「confusable / hard negative」=同一個概念，指「看起來很像但其實是錯解」的單。階段 0 由人工**先驗**標註一批陷阱單；階段 4 由「否決分因」產生的 hard negative 是**同一個資料池的延伸**（往池裡新增），不是另一套。負樣本三元組統一格式為 `(query, 正解單, 陷阱單)`。

## 你的角色

你是一位資深 RAG / 檢索系統工程師。你要為一套「AI 值班輔助系統」做**檢索層的準確度重構**。這不是從零開發，請務必先理解現況再動手，並嚴格按照下方的分階段順序推進，每一步都要用數字證明變準才能進下一步。

請先閱讀專案內的兩份設計文件再開始：
- `架構.md`
- `設計-檢索準確度重構.md`

> **順序權威聲明**：本執行 prompt 的「階段 0a → 階段 0 → 階段 1 → 2 → 3 → 4」為**唯一執行順序**。設計文件僅供背景理解；若兩者順序或顆粒度不一致，**一律以本檔為準**。

---

## 背景與現況

- **業務場景**：值班團隊面對 user 報案（JIRA ITSM，累積兩年資料），需快速找出解法。
- **現有系統**：已有一套 RAG。把過往資料整理成**關鍵字 index**，搭配 TKMS 的 SOP 提供給值班。UI 上有「採用 / 否決」按鈕，否決時請值班補充，並有 review 功能驗證答對與否。
- **已備妥的資產**：歷史資料已搭配「當時值班的回答」整理成知識庫（corpus）。**注意：這是 corpus（被搜尋的題庫），不是評測用的 golden set，golden set 仍需另外建（見階段 0）。**

## 核心痛點與目標

- **核心痛點**：準確度不夠高，最常見的失敗型態是「**完全找錯**」——撈回來的是不相干的案子。
- **根因**：對「模糊、用詞不一致」的文字報案，**純關鍵字檢索語意對不上**，召回大量無關案件。
- **北極星指標**：找得準（檢索命中品質）。

## 範圍與前提（務必遵守）

- **第一版範圍**：只處理**非 pipeline 的文字類報案**（權限、環境、咨詢等）。pipeline 失敗類本版不做，但要能辨識並擋下（判定規則見階段 1）。
- **知識來源權重**：**歷史已解案為主、SOP 為輔**。但在「正確性/時效」維度上有例外——當現行 SOP 與舊歷史解衝突時，以現行 SOP 為準（見階段 2 失效知識處理）。
- **不確定時的行為**：資訊不足時，AI 要提示值班「該追問 user 什麼」，**而非硬猜**。
- **介面**：獨立 Web 工具，不動 JIRA。
- **資料存取**：JIRA 與 TKMS 的取得路徑已有，本任務不涵蓋。
- **回饋飛輪**：採用/否決/補充/review 機制已存在，可作為評測與調校燃料（但須先過濾標註偏差，見階段 4）。

## 執行原則（最重要）

1. **不要一次全做。** 嚴格按階段 0a → 0 → 1 → 2 → 3 → 4 推進。
2. **每一步都對照 golden set 量測，指標確實提升才進下一步。**
3. **驗收門檻機制（解決「用數字當關卡卻沒數字」的矛盾）**：
   - 所有 gate 門檻、閾值、權重，**一律從 golden set 掃描（sweep）出來，禁止手填拍腦袋**。
   - 跑完 baseline 後，與使用者共同對每階段設定**具體 gate 數字**（如「Recall@10 相對 baseline 提升 ≥ N%」），寫入儀表板。
   - 下方各階段以 `【待填:XXX】` 標出必須先和使用者敲定的數字，**未設定者不得宣稱該階段通過**。
4. **寧可不答，不要亂答。** 資訊不足或候選不夠相關時，輸出「建議追問 user 的問題清單」，不要硬從爛候選裡挑一個。
5. **動手前先盤點現有程式碼與資料結構，回報你打算怎麼做，再實作。**

## 階段 0a：資料治理前置（在建 corpus / index 之前，最高優先）

> 兩年 ITSM 單裡必然散落明文密碼、token、內網 IP、AD 帳號、user 個資。若直接向量化、餵 LLM、句級引用原文，會把敏感資訊散播出去，甚至送出組織邊界——這是「一上線就被資安喊停」的風險，且補晚了整個 corpus 要重洗。

1. **Ingestion 遮罩**：corpus / index 建立前先過一層 PII / secret 偵測與遮罩（regex + token entropy 偵測 + 既有 DLP 規則），遮罩後才入庫。截圖（讀圖路線）OCR 後須走同一道遮罩。
2. **輸出側二次掃描**：生成草稿引用原文時，再掃一次，避免把密碼/token 貼回給值班。
3. **模型部署邊界**：明確界定哪些步驟可用外部 API、哪些必須內部自架。**含敏感資訊的內容（尤其讀圖）不得送出組織邊界**，除非已遮罩。
4. **存取控制與稽核**：corpus 查詢留稽核日誌（誰查到哪張單）。

**驗收**：抽樣確認遮罩生效（建議抽樣 **≥ 200 張或全量 5%，取大者**；敏感資訊召回為 0）；確認對外模型呼叫不含未遮罩敏感內容。
> 註：階段 0a 的待填值（抽樣量等）在**啟動時即與使用者敲定，不必等 baseline**（「跑完 baseline 才設 gate」只適用階段 0 之後的準確率門檻）。

## 階段 0：地基 —— 先建量測能力

**目標**：打造一把能抓到「完全找錯」的客觀尺。

1. **建分層 Golden Set（評測集）**
   - 從兩年歷史案挑出高信心案，做成「**查詢 → 正解單號[]**」對照表。
   - **雙軸分層**：報案類型（權限/環境/咨詢）× 難度（易/中/難）= 9 格。**目標總量 300~500 筆，每格盡量 ≥ 30**；若某格歷史案不足 30，記錄實際筆數並標為「低信心格」，**不阻擋階段 0 通過**，但該格指標需單獨標示「樣本不足」。
   - **困難負例標註（量「完全找錯」的關鍵）**：對每個「難」級 query，額外標 1~3 張陷阱單（見開頭名詞統一），存成 `query → {正解單號[], 陷阱單號[]}`。這批同時是階段 4 三元組的種子。
   - **query 端用「原始未整理的報案內文」**（模擬真實髒輸入），不要用整理過的乾淨文字，否則分數虛高。
   - **正解複核要驗「解法現在仍有效」**：不能只看「當年有沒有被採用」（採用 ≠ 正確，且可能已過時）。由值班複核「此解法在今天的環境是否仍正確」。
   - **複核品質保證（整套量測的地基，務必做）**：複核本身會錯（趕、偷懶、把「user 放棄了」誤當「解對了」），而錯誤一旦寫進 golden set 就被「數字化」成權威、比沒量測更危險。對策：(a)「難」格與「跨版本」格**強制雙人盲覆核、不一致才升級裁決**，記錄複核者 ID 與時間；(b) 每季由 owner **抽 5% 做「複核的複核」**，專抓「沒復發就當對」的假陽性；(c)「**未復發」要有明確定義與觀察窗**（多久沒回來算解決、同 user 換說法再報算不算復發），不可當空詞。注意：標註者間一致性高 ≠ 正確（可以一致地錯）。
   - **避開三個坑**：(a) 不可讓題目單自己命中自己（資料洩漏）——測「找到同類其他單」或把題目單暫排除於 corpus；(b) 一題允許對應多個正解；(c) 正解需人工複核。
   - **加一條難題分層**：「跨版本 / 已變更流程」的案（當年正確、今天已改），專門量測這類的找錯率。
   - **pipeline 漏接陷阱樣本**：放入一批「**pipeline 失敗但內文無 URL**」的單當已知陷阱（這類會被階段 1 規則漏接、當文字類硬走檢索而必然找錯），量測「該擋沒擋下」的漏接率。

2. **建指標儀表板**
   - 主指標 **Recall@k（k=5,10）**；reranking 上線後加 **MRR / Hit@1**。
   - **召回上限指標：「正解出現在 top-50（rerank 前）的比率」**——把「召回覆蓋率」與「排序品質」拆開。若這個數字低，問題在召回不在精排，別浪費力氣調 reranker（呼應階段 2 的召回保底）。
   - **對抗指標：「陷阱單擠進 top-1 / top-3 的比率」**（越低越好，比 Recall@k 更貼近「完全找錯」，是本專案最該盯的尺）。**此指標分母 = 有標陷阱單的 query 數（即難級子集）**，勿用全體當分母稀釋。
   - 所有指標**按分層拆開呈現**，避免大宗簡單案灌水。
   - gate 範例：Recall@10 相對提升 ≥ `【待填:N%】`、陷阱單 top-1 比率 ≤ `【待填:%】`。

3. **建全鏈路 Trace**：每次查詢一個 trace_id，貫穿「原始 ticket → 結構化查詢 → 召回候選(含各路分數) → rerank 前後 → 最終引用」，讓每個找錯案都能定位到責任層。

**驗收**：golden set 各 cell 達標或標記低信心；跑現有關鍵字流程拿到 baseline 數字。

## 階段 1：止血 —— 不確定就追問，別硬猜（低成本、高回報）

1. **輸入充足度閘門**（檢索前）：用 LLM 判斷此 ticket 資訊量是否足以檢索。不足時不硬撈，直接輸出「建議追問 user 的問題清單」，待補齊再查。
   - **此 query 理解 LLM 同時內含一個「結構化症狀萃取」子任務**（輸出固定 symptom schema），階段 1 即落地此子任務（此時主要用於充足度判斷）。**萃取 schema 須在進入階段 2 前凍結**，因為階段 2 會用「同一套 prompt/schema」對 corpus 端重萃取以保證對稱（見階段 2.3）。
2. **棄答門檻**（檢索後）：此階段混合檢索與 reranker 尚未上線，**門檻建立在現有關鍵字檢索分數之上**。用**絕對門檻**（非永遠取 top-k），連最好的候選都低於門檻就不生成答案、改追問。
   - **門檻怎麼定**：在 golden set 上掃描門檻值，要求「自信找錯率」相對 baseline 下降 ≥ `【待填:X%】`、且誤拒率（該答卻棄答）≤ `【待填:Y%】`；取滿足條件的**最低**門檻。禁止手填。
3. **pipeline 類擋下（最小規則）**：判定 = 報案內文含可辨識的 pipeline URL（**純規則比對即可，不需 LLM、不需串下游、不需建分類模型**）。命中則直接回固定訊息「此為 pipeline 失敗類，本版不處理」並終止，不進檢索。這就是「擋下」的全部要求。
   - **已知限制**：判定**僅掃文字內文**，截圖內的 pipeline URL 不在本版偵測範圍（無 URL 的 pipeline 單會漏接，由階段 0 漏接率指標盯著）。
   - **被擋下的單留一筆輕量日誌**（命中規則、時間，純計數即可），供第二版評估「範圍外需求量」用，避免變成看不見的服務黑洞。

**驗收**：「自信地找錯」比例下降達門檻；誤拒率在約定範圍內。

## 階段 2：打根因 —— 換掉純關鍵字檢索（治本）

1. **混合檢索（Hybrid）+ 召回覆蓋保底（打「正解從未進候選集」這種完全找錯）**：語意向量（embedding）+ 關鍵字 BM25 同時召回，用 **RRF** 融合。向量處理「用詞不同意思相同」，BM25 對錯誤碼/服務名做精確命中保護。
   - **多查詢召回（multi-query fan-out）**：用階段 1 已在跑的查詢理解 LLM，順手產出 2~3 個改寫變體（同義說法、症狀導向 vs 服務導向），各自召回後**併集再進 RRF**。復用既有 LLM 呼叫、零新模型，專打「用詞不一致導致正解沒被召回」。
   - **原文保命路**：除了萃取後的結構化句，**保留一路用「原始 query 文字」的獨立 BM25 召回**並聯進 RRF。如此即使萃取 LLM 抽錯（query/document 兩端會對稱地錯向同一方向），原文 BM25 仍能保底，不把雞蛋全押在萃取這顆 LLM。
   - RRF 的 k 常數、各路權重、召回 top-N，**在 golden set 上做小範圍 sweep 決定**，記錄於實驗紀錄，禁止手填。
2. **中文/技術術語 Embedding 選型 + 領域詞典**：報案是「中文描述 + 英文服務名/錯誤碼」混雜，選中英混雜適配好的模型（如 BGE-M3 一類）。BM25 端中文先斷詞並掛領域詞典——這是精確命中的**保命線**，不是一句帶過：
   - 領域詞典**不手刻**：從現有 CI/CD 平台清單（Gitlab/Nexus/Harbor/Azure 的 repo/專案/群組名）與 ITSM 既有標籤批次匯入當種子。
   - 錯誤碼 / URL / 群組名等**強結構 token 用 regex 先保護性切出**再送斷詞器（可與階段 0a 的 secret 掃描共用同一道 pipeline）。
   - **alias map（服務改名/遷移的單點解，重要）**：CI/CD 平台會改名、遷移（Harbor 換 registry、Azure 訂閱重組），導致一整批 corpus 同時系統性偏移、黑名單逐筆標不完。維護一份**服務名/別名映射表**，同時掛在 BM25 領域詞典與查詢理解同義詞擴展兩端，集中管理改名。
3. **問題/解法分離 + query/document 對稱萃取（本階段增益最大，務必做對）**：
   - 歷史單拆成 `role=problem`（症狀）與 `role=solution`（解法）兩種 chunk，檢索只比對 problem chunk（以問搜問），命中後帶出 solution。
   - **關鍵：corpus 端的 problem 也要用階段 1 凍結的同一套萃取 schema/prompt 萃取成「結構化症狀句」再 embedding**，保證 query 端與 document 端**對稱**——否則「以問搜問」不成立。
   - **萃取版本管理**：corpus chunk 一併記 `extraction_model_version` + `extraction_prompt_version`（與 `embedding_version` 不同概念，須分開記）。query 端換版時與 corpus 端比對，不一致則**告警並排程回填重萃**（不必即時，但落差要看得見，否則「對稱」會悄悄失效且無告警）。ingestion 時對萃取結果做**輕量抽樣品檢**（抽 N 筆人看症狀句有沒有抽歪），納入驗收。
   - **chunk 粒度**：problem chunk 控制在約 256~512 token，超長按語意切分；log dump / 簽核樣板等雜訊**可進 BM25 但不進向量**，避免稀釋語意中心。
   - trace 記錄「萃取後的結構化症狀句」，讓找錯案能一眼看出是否萃取層的鍋。
4. **雙 corpus 主從融合**：歷史案、SOP 分開索引，歷史案加權 + 保底配額。
5. **失效知識處理（時效污染，提前到本階段）**：
   - 索引帶 `resolved_at`，檢索/rerank 期對舊案做**溫和時間衰減**（當 feature，不硬切，避免誤殺仍有效的老解法）。
   - **人工失效黑名單**：值班於 review 時可一鍵標記「此解法已過時/已失效」，命中即排除於召回。利用既有 review 動線、零額外建設，比純演算法降權可靠。
   - **黑名單治理（重要：它是繞過所有數字 gate 的人工旁路，誤標會把高頻正解整批逐出候選集 → 製造「完全找錯」，且 CI 回歸/漂移監控抓不到）**：每次標記須附 (a) **標記者 ID + 失效理由 + 生效範圍**（全域／特定環境或版本）；(b) **爭議升級**——同一解法被標失效又被他人主張有效時，走與階段 0「難格雙人盲覆核」**同一條裁決動線**由 owner 拍板，不容單人靜默翻轉；(c) 黑名單每次增刪當作一次「**corpus 變更事件**」納入階段 4 漂移對賬，讓「人工排除導致的線上否決率變化」看得見。三點皆複用既有機制、零新建設。
6. **Metadata 軟過濾**：按類別/已解狀態先過濾去噪，分類信心低時放寬，避免誤殺正解。

**驗收**：Recall@k 明顯提升、陷阱單 top-1 比率下降；特別看「模糊/用詞不一致」與「跨版本」兩層。

## 階段 3：精排 —— 把最該用的推到最前

1. **兩階段 reranker**：
   - 第一階 cross-encoder（如 bge-reranker，支援中文）對 hybrid 召回 top-50 精排取 top-8。
   - 第二階 LLM 對 top-8 做相關性裁決（relevant: yes/partial/no + 理由），no 丟棄。**理由欄位可直接露出給值班當「為何推薦這張單」**，提升信任。
2. **信心分數校準**（資料夠後再做）：用 Platt scaling / isotonic regression 把 reranker 原始分轉成真實命中機率，對外**誠實顯示不確定**、寧可保守。
3. **低信心轉定向追問**：由候選間的差異反推「需要 user 補充哪個欄位」，與階段 1 充足度檢查形成下游兜底。

**驗收**：MRR / Hit@1 提升；top-1 為無關案（含陷阱單）的比例下降。

---

## 階段 4：生成可信度 + 飛輪 —— 讓系統越用越準（且不被自己的偏差帶歪）

1. **句級引用 grounding**：答案每段標清來源 + 信心度，並做 grounding 驗證（答案句是否被引用來源蘊含），降低幻覺。低信心改追問。
2. **回饋飛輪標註去偏（關鍵，否則飛輪會自我強化偏差）**：
   - **區分強/弱信號**：golden set 與正樣本**只收強信號**（review 複核答對 + 問題確實解決未復發），不收「只是被點採用」的弱信號（採用 ≠ 正確）。
   - **打破位置偏差**：值班幾乎只採用 top-1~3，飛輪只會強化既有偏好、看不到「系統根本沒撈到的正解」。引入**少量隨機曝光 / 反事實記錄**（記下沒被採用但其實相關的候選）。
   - **否決要分因**：只有「不相關」才當 hard negative；「相關但值班想自己處理」不可當負樣本，否則教壞模型。
   - 監控標註者間一致性，避免單一重度標註者主導。
3. **增量索引 + 知識新鮮度**：新解決案件即時進 index（先過階段 0a 遮罩），帶 `resolved_at` 與 embedding 版本。
4. **漂移監控（防 golden set 給假安全感）**：golden set **每季滾動更新**納入最新案；常設監控「線上 query 分布 vs golden set 分布」與「線下 Recall vs 線上否決率」對賬，**一旦脫鉤示警**。對「查無正解」類（新問題、corpus 沒有）單獨追蹤。
   - **實體/服務名分布漂移偵測**：當線上 query 頻繁出現 corpus 幾乎沒有的新服務名時示警——這通常代表「服務改名 / 新系統上線」，提示更新階段 2 的 alias map 與領域詞典。
5. **CI 自動回歸**：每次改任一層自動跑全 golden set + gate 判定，不達標擋發布，gate 線為 `【待填】`。

**驗收**：每輪微調後在固定 golden set 上不退步；線下 Recall 提升能對應線上否決率下降。

## 維運與信任（上線後常設，非一次性）

- **延遲預算**：查詢理解 + 讀圖 + hybrid + cross-encoder + LLM 裁決串起來可能數秒~十幾秒，**救火時慢 = 不用**。訂單次查詢 **P95 延遲預算 `【待填:N秒】`**，並把 LLM rerank / 讀圖 / grounding 設為**可降級**（高負載退回 cross-encoder only）。
- **成本上限**：每張單多次 LLM 呼叫，量大會失控。訂成本上限，且禁止為省錢偷偷降級模型導致準確率悄悄下滑。
- **信任健康度**：追蹤**採用率隨時間的曲線**（值班被「自信找錯」坑幾次就會永久棄用），作為比單點準確率更真實的健康指標。
