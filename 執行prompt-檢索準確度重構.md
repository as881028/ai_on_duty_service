# 任務 Prompt：AI 值班輔助系統 — 檢索準確度重構

## 你的角色

你是一位資深 RAG / 檢索系統工程師。你要為一套「AI 值班輔助系統」做**檢索層的準確度重構**。這不是從零開發，請務必先理解現況再動手，並嚴格按照下方的分階段順序推進，每一步都要用數字證明變準才能進下一步。

請先閱讀專案內的兩份設計文件再開始：
- `架構.md`
- `設計-檢索準確度重構.md`

---

## 背景與現況

- **業務場景**：值班團隊面對 user 報案（JIRA ITSM，累積兩年資料），需快速找出解法。
- **現有系統**：已有一套 RAG。把過往資料整理成**關鍵字 index**，搭配 TKMS 的 SOP 提供給值班。UI 上有「採用 / 否決」按鈕，否決時請值班補充，並有 review 功能驗證答對與否。
- **已備妥的資產**：歷史資料已搭配「當時值班的回答」整理成知識庫（corpus）。**注意：這是 corpus（被搜尋的題庫），不是評測用的 golden set，golden set 仍需另外建（見階段 0）。**

## 核心痛點與目標

- **核心痛點**：準確度不夠高，最常見的失敗型態是「**完全找錯**」——撈回來的是不相干的案子。
- **根因**：對「模糊、用詞不一致」的文字報案，**純關鍵字檢索語意對不上**，召回大量無關案件。
- **北極星指標**：找得準（檢索命中品質）。

## 範圍與前提（務必遵守）

- **第一版範圍**：只處理**非 pipeline 的文字類報案**（權限、環境、咨詢等）。pipeline 失敗類（有 URL、可串 Nexus/Harbor/Azure）本版不做，但要能辨識並擋下，不可混進文字檢索鏈。
- **知識來源權重**：**歷史已解案為主、SOP 為輔**。很多解法是值班的隱性經驗，只存在於過往單子。
- **不確定時的行為**：資訊不足時，AI 要提示值班「該追問 user 什麼」，**而非硬猜**。
- **介面**：獨立 Web 工具，不動 JIRA。
- **資料存取**：JIRA 與 TKMS 的取得路徑已有，本任務不涵蓋。
- **回饋飛輪**：採用/否決/補充/review 機制已存在，可作為評測與調校燃料。

## 執行原則（最重要）

1. **不要一次全做。** 嚴格按階段 0 → 4 推進。
2. **每一步都對照 golden set 量測，Recall 等指標確實提升才進下一步。** 否則就是「感覺不準但不知道改哪」。
3. **寧可不答，不要亂答。** 資訊不足或候選不夠相關時，輸出「建議追問 user 的問題清單」，不要硬從爛候選裡挑一個。
4. 動手前先盤點現有程式碼與資料結構，回報你打算怎麼做，再實作。

---

## 階段 0：地基 —— 先建量測能力（最高優先，其他都依賴它）

**目標**：打造一把客觀的尺，否則後續每步都是瞎改。

1. **建分層 Golden Set（評測集）**
   - 從兩年歷史案挑出「有 review 答對 + 有採用紀錄」的高信心案，做成「**查詢 → 正解是哪幾張單**」的對照表。
   - **雙軸分層**：報案類型（權限/環境/咨詢）× 難度（易/中/難）。每格至少 30 筆，先做 300~500 筆 MVP。
   - **query 端用「原始未整理的報案內文」**（模擬真實髒輸入），不要用整理過的乾淨文字，否則分數虛高、測不出真實痛點。
   - **避開三個坑**：(a) 不可讓題目單自己命中自己（資料洩漏）——測「找到同類其他單」或把題目單暫排除於 corpus；(b) 一題允許對應多個正解單號；(c) 正解需由值班人工複核。
2. **建指標儀表板**：主指標 **Recall@k（k=5,10）**；reranking 上線後加看 **MRR / Hit@1**；輔以否決率/採用率。**每個指標都按分層拆開呈現**，避免大宗簡單案灌水。
3. **建全鏈路 Trace**：每次查詢一個 trace_id，貫穿「原始 ticket → 結構化查詢 → 召回候選(含各路分數) → rerank 前後 → 最終引用」，讓每個找錯案都能定位到責任層。

**驗收**：golden set 各 cell 達最低筆數；跑現有關鍵字流程拿到 baseline 數字。

---

## 階段 1：止血 —— 不確定就追問，別硬猜（低成本、高回報、不依賴進階技術）

1. **輸入充足度閘門**（檢索前）：用 LLM 判斷此 ticket 資訊量是否足以檢索。不足時不硬撈，直接輸出「建議追問 user 的問題清單」，待補齊再查。
2. **棄答門檻**（檢索後）：用**絕對相關性門檻**（非永遠取 top-k）。連最好的候選都低於門檻就不生成答案，改追問。把「沒有好答案」變成系統的一級輸出。

**驗收**：在 golden set 上，「自信地找錯」的比例顯著下降；同時控制「該答卻棄答」的誤拒率在可接受範圍。

---

## 階段 2：打根因 —— 換掉純關鍵字檢索（治本）

1. **混合檢索（Hybrid）**：語意向量（embedding）+ 關鍵字 BM25 同時召回，用 **RRF** 融合（免去跨量綱調權重）。向量處理「用詞不同意思相同」，BM25 對錯誤碼/服務名做精確命中保護。
2. **中文/技術術語 Embedding 選型**：報案是「中文描述 + 英文服務名/錯誤碼」混雜，選中英混雜適配好的模型（如 BGE-M3 一類）；BM25 端中文先斷詞並掛領域詞典（避免 Gitlab/Nexus/Harbor/Azure 等專名被切碎）。
3. **問題/解法分離 Chunking（增益最大）**：歷史單拆成 `role=problem`（症狀/報案內文）與 `role=solution`（值班解法）兩種 chunk。**檢索只比對 problem chunk（以問搜問）**，命中後再帶出對應 solution。避免解法文字污染語意中心。
4. **雙 corpus 主從融合**：歷史案、SOP 分開索引，歷史案加權 + 保底配額，避免措辭工整的 SOP 洗掉真正有用的隱性經驗。
5. **Metadata 軟過濾**：按類別/已解狀態先過濾去噪，但分類信心低時放寬，避免誤殺正解。

**驗收**：對照 baseline，Recall@k 明顯提升，「完全找錯」比例下降；特別看「模糊/用詞不一致」那層的提升。

---

## 階段 3：精排 —— 把最該用的推到最前

1. **兩階段 reranker**：
   - 第一階用 cross-encoder（如 bge-reranker，多語支援中文）對 hybrid 召回的 top-50 精排，取 top-8。
   - 第二階用 LLM 對 top-8 做相關性裁決（輸出 relevant: yes/partial/no + 理由），把 no 丟棄不進生成。
2. **信心分數校準**（資料夠後再做）：用 Platt scaling / isotonic regression 把 reranker 原始分轉成真實命中機率，讓顯示給值班的「信心 78%」真的對應約 78% 命中率。
3. **低信心轉定向追問**：由候選間的差異反推「需要 user 補充哪個欄位」，與階段 1 的充足度檢查形成下游兜底。

**驗收**：MRR / Hit@1 提升；top-1 為無關案的比例下降。

---

## 階段 4：生成可信度 + 飛輪 —— 讓系統越用越準

1. **句級引用 grounding**：答案每段標清來源（哪張歷史單 / 哪份 SOP）+ 信心度，並做 grounding 驗證（檢查答案句是否被引用來源蘊含），降低幻覺。低信心改追問。
2. **否決/補充轉負樣本回饋**：把既有「否決」案當 hard negative 餵回 reranker 微調；「補充」的正解回填 golden set。組成 (query, 正解, 被否決負解) 三元組。注意用時間切分防資料洩漏。
3. **增量索引 + 知識新鮮度**：新解決案件即時進 index，帶 `resolved_at` 與 embedding 版本；對過舊可能失效的解法做時效降權。
4. **CI 自動回歸**：每次改任一層，自動跑全 golden set + gate 判定，不達標擋發布。把「用數字證明變準」變成強制流程。

**驗收**：回饋飛輪每輪微調後在固定 golden set 上不退步；線下 Recall 提升能對應到線上否決率下降。

## 交付要求

1. 每完成一個階段，提交：**改了什麼、golden set 指標前後對比數字、trace 範例**。
2. 任何一步若指標未提升或退步，**停下來分析原因**，不要硬進下一階段。
3. 全程繁體中文回報。
4. 程式碼風格對齊既有專案。

## 本版不做（先記著，別誤入範圍）

- Pipeline 失敗類報案：解析 pipeline URL → 抓失敗原因 → 串 Nexus/Harbor/Azure 下游狀態。
- 兩類報案的自動分流器（本版只需「辨識並擋下 pipeline 類」）。
