# 人員協作指南 — 人能為「值班準確度」做什麼

> 系統面（embedding / BM25 / rerank / golden set / 飛輪 / 失效黑名單 / alias map / 敏感資訊遮罩）已經過 4 輪迭代收斂（見 `迭代日誌.md`）。
> 本文回答另一個維度的問題：**「人」（值班、知識維護者/管理員、主管）能做什麼，讓系統更準地解決值班問題。**

## 核心觀念：系統是放大器，不是發電機

這套 RAG 的「燃料」幾乎全來自人的日常動作。所有自動化機制（hybrid 檢索、rerank、飛輪）都是**放大器**——人餵進去什麼品質，系統就放大什麼品質。設計文件最關鍵那句：**「系統越準，散播錯誤解法越快」**。所以人最大的價值不是「用得多」，而是**把對的訊號餵準、把錯的訊號擋住**。

下面按三個角色給出「該做的事」與「不該做的事」，並標注對應到設計裡的哪個機制。

---

## 一、第一線值班人員

第一線的貢獻集中在**書寫紀律**與**按鈕的誠實使用**，多數零建設、直接見效。

### 低成本高回報、今天就能做（純習慣）

1. **否決時先選原因，再補一句正解** — 誠實區分「(A) 撈回來根本不相關（找錯）」 vs 「(B) 相關，但我想自己處理 / 情境特殊」。A 類盡量補正解單號或方向。
   - 為何有效：只有 A 類能當 **hard negative（陷阱單）** 餵回 reranker；B 類若被誤當負樣本會**教壞模型**（把有效解打成錯解）。對應〔階段 4.2 否決分因〕。
2. **截圖裡的錯誤訊息打成文字貼回** — 別只丟一張圖。
   - 為何有效：錯誤碼是 **BM25 精確命中的保命線**；鎖在圖裡 = 製造「完全找錯」的典型上游成因。對應〔階段 1 讀圖降級邏輯〕。
3. **系統提示「資訊不足、建議追問」時，照著問 user，別硬猜硬撈** — 不要繞過充足度閘門硬塞模糊內文。
   - 為何有效：爛 query 進＝爛結果出，這是「完全找錯」最上游元兇。對應〔階段 1 充足度閘門 / 棄答門檻〕。
4. **症狀導向書寫、用正式服務名（少用個人黑話縮寫）** — 寫清「症狀＋服務/環境/錯誤碼＋解法」。
   - 為何有效：corpus 端解法未來會被同一套 schema 萃取成症狀句來「以問搜問」，書寫越貼近症狀結構，**對稱萃取**越準。對應〔階段 2.3 對稱萃取、2.2 alias map〕。

### 需要流程/工具配套才能做好

5. **採用後等「真的解決且未復發」才在 review 標答對** — 不是一點採用就結案。
   - 為何有效：golden set 只收「review 複核＋未復發」的**強信號**；「採用 ≠ 正確」。需先有「未復發」的明確定義與觀察窗。對應〔階段 0.1、4.2〕。
6. **發現舊解法現在會出事 → 一鍵標失效黑名單，並寫清失效範圍與理由** — 範圍是全域還是僅特定環境/版本。
   - 為何有效：rerank 會非常「有效地」把語意最像、看起來最對的埋雷舊解推到 top-1。需有 owner 爭議裁決動線，否則單人靜默翻轉會把高頻正解整批逐出。對應〔跨層治理 B、階段 2.5〕。

### 值班的反模式（最易破壞準確率）

- **否決即丟**（不選原因不補正解）→ 污染 hard negative 池，**最高危**，因為它偽裝成「我有給回饋」。
- **採用即結案** → 弱信號灌進 golden set，評測尺本身腐化。
- **只看 top-1~3 的位置偏差** → 飛輪只強化既有偏好、看不到沒撈到的正解。偶爾往下多看幾筆。
- **無視追問、硬撈一個交差** → 親手製造「自信地找錯」。
- **黑名單亂標 / 標了不寫範圍** → 把「我這次環境特殊」誤標成「全域失效」，繞過所有數字 gate。

### 不該強加給值班（過度負擔、低 ROI）

- 手填完整結構化 schema（逐欄填症狀/實體/錯誤碼）→ 那是查詢理解 LLM 的工作；值班只要「症狀導向自然書寫」。
- 做 PII/secret 遮罩 → ingestion pipeline 的自動職責，人工必漏。
- 主動標困難負例三元組 → 標註團隊的專門工作；值班的貢獻是「否決分因」自然產生 hard negative 種子。
- 理解 Recall@k / MRR / RRF → 與第一線無關，只增加認知負擔。

## 二、知識維護者 / 系統管理員

管理員的人力應集中在**「機器判斷不了的價值裁決」**；所有「偵測、記錄、跑分、遮罩掃描、調閾值」都該是自動 pipeline，人只接異常告警。

### 不做就會慢慢腐蝕準確率的「必備治理」（按腐蝕速度與隱蔽性排序）

1. **飛輪標註去偏（最陰險）** — 監控標註者一致性，避免單一重度標註者主導；抽查否決分因是否正確、採用是否為強信號。〔頻率：每月看一致性、每週抽查否決分因〕
   - 注意陷阱：一致性高 ≠ 正確（可以一致地錯），須配合「複核的複核」一起看。對應〔階段 4.2、階段 0〕。
2. **失效黑名單治理 + 衝突裁決** — 審核值班標記（須附 ID/理由/範圍）；同一解被標失效又被主張有效時走 owner 拍板；SOP 與舊歷史解衝突以 SOP 為準。〔標記事件觸發；每週批次複核；爭議即時〕
   - 警惕：黑名單是繞過所有數字 gate 的人工旁路，誤標會把高頻正解整批逐出且 CI/漂移監控抓不到，**「審核」本身不可省**。對應〔跨層治理 B、階段 2.5〕。
3. **Golden Set 季度滾動策展 + 複核的複核** — 難格/跨版本格強制雙人盲覆核；owner 每季抽 5% 抓「沒復發就當對」的假陽性；維護「未復發」定義與觀察窗。〔每季〕
   - 為何有效：golden set 是整套量測的地基，地基髒，後面每層「用數字證明變準」全是假的。對應〔階段 0、跨層治理 D〕。
4. **Alias Map / 領域詞典維護** — 服務改名/遷移（Harbor 換 registry、Azure 訂閱重組）的映射表，掛在 BM25 詞典與同義詞擴展兩端；接 4.4 漂移告警後更新。〔事件觸發＋每季盤點〕
   - 為何有效：平台改名讓一整批 corpus 同時系統性偏移，逐筆黑名單永遠標不完；這是少數「一次維護、整批受惠」的高槓桿動作。對應〔階段 2.2、4.4〕。

### 高價值但非「防腐蝕必備」

- **隱性經驗顯性化成 SOP** — 把只存在某張老單、沒進任何 SOP 的高頻有效解，萃取成正式 SOP 補進 TKMS。可享「SOP 為準」的時效正確性，也讓衝突裁決有權威依據；老人離職不致流失。〔每季主題式盤點〕

### 應自動化、不該變人工常設負擔

| 項目 | 判斷 |
|------|------|
| PII/Secret 遮罩逐筆人工檢查 | 自動 pipeline；人只做上線抽樣驗收（≥200 張或 5%）＋規則更新 |
| Corpus 變更事件記錄（黑名單增刪/增量索引/遮罩規則變更） | 系統自動寫事件日誌＋算對賬曲線；人只在曲線異常時介入 |
| 實體/服務名分布漂移偵測 | 自動偵測；人只「收到告警後更新 alias map」 |
| 每日人工跑 golden set / 看指標 | CI 自動回歸已每次改動跑＋gate 擋發布；季度策展＋告警驅動即可 |
| 正解全量雙人盲覆核 | 只要求難格/跨版本格；易格單人＋抽查 |
| 棄答/召回門檻人工調參 | 一律從 golden set sweep，禁止手填拍腦袋 |

## 三、主管 / 管理層

主管最該做的不是「管系統」，而是**「管誘因與信任邊界」**——這幾件事工程團隊無權自決，卻決定飛輪燃料的品質與系統準確率的長期上限。

1. **指定 Golden Set Owner 並授權其裁決動線** — 任命資深、被信服的值班當 owner，授權他主導複核裁決、黑名單爭議拍板、每季複核的複核，並給**明確時間配額**（非「有空再做」）。對應〔階段 0、2.5〕。
2. **把「棄答/低信心追問」明訂為正確行為，不算值班失職** — 制度白紙黑字：AI 棄答是設計預期的安全行為，不計個人 KPI 失分；「AI 自信給錯解導致事故」才需回溯。對應〔階段 1 棄答門檻、執行原則「寧可不答」〕。
3. **保護值班的標註時間，切斷「採用率＝KPI」的連結** — 排班工時切出受保護的「複核/標註」時間，績效認列**標註品質而非數量/速度**；明令採用率不得作為個人 KPI。對應〔階段 4 標註去偏、信任健康度〕。
4. **拍板資料治理與模型邊界合規** — 連同資安/法務正式拍板：(a) 哪些步驟可用外部 API、哪些須內部自架；(b) 含敏感資訊（尤其讀圖）不出組織邊界的紅線；(c) 查詢稽核日誌權責。並授權「治理未過前不准建 corpus」。對應〔階段 0a，ingestion 前置、最高優先〕。
5. **取得跨團隊資料授權與「現行 SOP 為準」的組織背書** — 向 JIRA/TKMS owner 取得正式資料權與更新承諾；在組織層級確立「SOP 與舊歷史解衝突時以 SOP 為準」。對應〔階段 2 失效知識處理〕。
6. **共同敲定 gate 數字、明訂擋發布決策權、建立事故回溯機制** — 與工程定 `【待填】` 各階段 gate（Recall 提升、陷阱單 top-1 上限、棄答誤拒率、P95、成本）；事故走 trace_id 回溯定責到「責任層」而非究責值班個人，讓人敢誠實上報找錯案。對應〔階段 0 trace、執行原則門檻機制、階段 4 CI gate〕。

### 誘因設計錯誤會反咬準確率（務必避開）

1. **採用率當個人 KPI（最致命）** — 誘發刷數字、弱信號灌爆飛輪，連「信任健康度曲線」這個監控儀表本身都被污染。
2. **AI 棄答當 KPI 失分** — 逼工程調低棄答門檻硬撈，拆掉階段 1 安全網，「完全找錯」回升。
3. **平均處理時間當唯一績效** — 誘發趕時間亂按、複核草率。
4. **黑名單標記數當積極度指標** — 誘發濫標，誤標把高頻正解整批逐出。黑名單**重質不重量**。
5. **究責「找錯案」本身** — 讓值班隱瞞找錯、不敢否決，斷掉負樣本來源。該回溯的是「自信給錯且未標記」，不是「系統找錯被人即時攔下」。

### 主管不該做（官僚化、低 ROI）

- 親自審每張黑名單/每筆複核（owner＋抽樣的事，微觀管理會成瓶頸）。
- 要求工程每週做正式 gate 簡報/繁複審批（已有 CI＋儀表板＋trace，看告警即可）。
- 把標註者一致性 KPI 化（會誘導「一致地錯」；當異常訊號看即可）。
- 把讀圖上線時程當必達里程碑施壓（讀圖是加分項、有充足度檢查兜底，會排擠階段 1 高回報項）。
- 為棄答案件建繁複事後檢討會（棄答是預期行為，看趨勢、異常才查）。

## 跨角色共識：四條紅線（三個視角獨立得出的同一結論）

這四條由值班、管理員、主管三個視角**各自獨立**指向同一處，是訊號最強、最不該妥協的地方：

1. **採用率永遠不能當個人 KPI** — 只能當系統整體的「信任健康度曲線」。一旦 KPI 化，同時毀掉飛輪燃料品質與監控儀表。
2. **棄答/追問是正當行為，不是系統爛** — 上到制度、下到按鈕使用都要守住；它是「完全找錯」最便宜有效的安全網。
3. **否決必須分因** — 「找錯」才是 hard negative，「想自己處理」不是。混為一談會教壞 reranker，這是最便宜卻最常被忽略的一步。
4. **失效黑名單必須有 owner 與裁決動線** — 它是繞過所有數字 gate 的人工旁路，威力最大、誤用代價也最大（高頻正解整批逐出且監控抓不到）。

## 一句話總結

工程把**放大器**調到最好；人決定**餵什麼進去**。值班守四個書寫/按鈕習慣，管理員守四項防腐蝕治理、把其餘交給自動 pipeline，主管守誘因與信任邊界——三者合起來，才是系統長期準確率的真正上限。
